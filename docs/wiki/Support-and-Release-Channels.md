# Support and Release Channels

Re:T-UI is available through more than one channel, but those channels are not meant for the same kind of user.

This page explains what each one is for and what kind of support you should expect.

## Official Channels

### Play Store

The Play Store build is the official consumer release.

This is the best choice if you want:

- the easiest install path
- straightforward updates
- the most polished stable release
- the official way to support development

This is the version normal users should buy and install.

### Firebase App Distribution

Firebase is the invite-based beta and testing channel.

This is the right place for:

- preview builds
- testing fixes before wider release
- device-specific validation
- fast iteration with active testers

Firebase builds are official, but they are not the same promise as the Play Store build.

The testing invite is shared with active testers when a preview build needs validation.

### GitHub

GitHub is the project home for:

- source code
- wiki/docs
- issue tracking
- community visibility
- self-built APK workflows

GitHub is public because Re:T-UI benefits from openness, testing, and trust.

## Support Policy

### Fully Supported

- official Play Store builds

### Supported on a testing basis

- official Firebase builds

These builds are part of the development process, but they may be rougher and may change more often.

### Community / best-effort only

- self-built APKs
- modified builds
- third-party forks

That does not mean those builds are unwelcome. It just means they are not the default support target.

## Why Keep GitHub Public?

Because Re:T-UI is the kind of project that benefits from:

- visible progress
- transparent development
- community experimentation
- easier bug reporting
- long-term trust

The public repo is not a problem by itself. The important thing is keeping the official release path clear.

## What Users Are Paying For

People buying Re:T-UI on the Play Store are not paying for secret source code.

They are paying for:

- convenience
- trust
- stable updates
- the official release path
- supporting continued development

That is a healthy model for this kind of app.

## Simple Rule of Thumb

- want the cleanest normal-user experience: use Play Store
- want to test and help shape upcoming builds: use Firebase
- want to inspect, build, or modify the project: use GitHub

## Maintainer Guidance

To keep this sustainable, Re:T-UI should always communicate clearly:

- which build is stable
- which build is beta
- which builds are unofficial

The more obvious the channel boundaries are, the easier it is to keep support manageable.
